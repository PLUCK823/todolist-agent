## 1. 提示词架构

Agent 的提示词由三层组成：

```text
┌──────────────────────────────────┐
│        系统提示词 (System)        │  ← 角色定义 + 能力说明 + 行为规则
├──────────────────────────────────┤
│      工具描述 (Tool Schema)       │  ← 每个函数的名称、参数、docstring
├──────────────────────────────────┤
│       对话上下文 (History)        │  ← 最近 N 轮对话记录
└──────────────────────────────────┘
```

- **系统提示词** 定义 Agent 的身份和能力边界（只能操作待办，不能做其他事）
- **工具描述** 由 LangChain 从函数签名和 docstring 自动生成，因此 docstring 质量直接影响模型调用准确性
- **对话上下文** 压缩保留最近 10 轮，超出部分做摘要

## 2. 系统提示词（当前版本）

### 版本：v1.0

```text
你是一个待办事项管理助手。你可以帮助用户创建、查询、更新和删除待办事项。

## 你的能力
- 创建待办：理解用户提到的任务和属性（优先级、截止日期等）
- 查询待办：列出用户的待办，支持按状态和优先级筛选
- 更新待办：修改待办的标题、优先级、截止日期等
- 标记完成：将待办标记为已完成
- 删除待办：删除指定的待办

## 规则
1. 始终用中文与用户交流
2. 在执行操作前，向用户确认关键信息
3. 如果用户没有指定优先级，默认使用 medium
4. 操作完成后，用简洁的语言告知结果
5. 如果用户的请求不明确，主动询问缺失的信息

## 示例对话

用户："帮我创建一个待办"
助手："好的，请问待办的标题是什么？是否需要设置优先级或截止日期？"

用户："买牛奶，高优先级"
助手：调用 create_todo(title="买牛奶", priority="high")
助手："已为你创建高优先级待办「买牛奶」"

## 边界
- 你只能操作待办事项，不能执行其他类型的操作
- 不能浏览网页、发送邮件、访问文件系统
- 如果用户请求超出你的能力范围，礼貌说明并拒绝
```

### 设计原则

| 原则 | 说明 |
| ------ | ------ |
| **明确边界** | 清楚告知模型能做什么、不能做什么，防止幻觉 |
| **行为规则** | 用编号列表写规则，模型更容易遵循 |
| **示例驱动** | 提供 1-2 个 Few-shot 示例，比纯规则更有效 |
| **中文优先** | 用户群体是中文用户，不切换语言 |
| **确认机制** | 关键操作（删除等）需确认，避免误操作 |

## 3. 工具定义规范

### 3.1 Docstring 模板

每个工具函数的 docstring 必须包含以下要素：

```python
@tool
def create_todo(title: str, priority: str = "medium") -> dict:
    """一句话概述这个工具做什么。

    Args:
        title: 参数说明，包含取值范围或格式要求
        priority: 参数说明，列出可选值

    Returns:
        返回值说明

    Note:
        使用注意事项（可选）
    """
```

### 3.2 参数类型提示要求

```python
# ✅ 好：类型明确，LLM 能准确推断参数格式
def list_todos(
    priority: Optional[Literal["high", "medium", "low"]] = None,
    completed: Optional[bool] = None,
    limit: int = 20,
) -> list[dict]:
    ...

# ❌ 差：类型模糊，LLM 不知道传什么
def list_todos(filter=None):
    ...
```

### 3.3 工具函数清单

| 工具名 | 用途 | 参数 |
| ------ | ------ | ------ |
| `create_todo` | 创建待办 | title*, priority, description, due_date |
| `list_todos` | 查询待办列表 | priority, completed, limit |
| `get_todo` | 获取单个待办详情 | todo_id* |
| `update_todo` | 更新待办 | todo_id*, title, priority, description, due_date |
| `complete_todo` | 标记完成 | todo_id* |
| `delete_todo` | 删除待办 | todo_id* |

> `*` 表示必填参数。

## 4. 调优指南

### 4.1 常见问题与修复

| 问题 | 现象 | 修复方向 |
| ------ | ------ | ------ |
| 工具调用参数错误 | LLM 对参数名理解偏差，传错字段 | 优化 docstring 示例，增加参数别名处理 |
| 过度确认 | 每个操作都反复问用户，体验差 | 规则中加「简单操作直接执行」 |
| 中文 prompt 注入 | 用户在待办标题中写恶意 prompt | 待办标题作为数据传递，不拼入 prompt 模板 |
| 上下文遗忘 | 长对话中忘记之前的待办 | 增大历史轮数，或对历史做摘要压缩 |
| 工具选择错误 | 更新操作误调用创建工具 | 在 docstring 中增加负面示例 |

### 4.2 测试方法

```bash
# 运行 Agent 测试套件
cd agent-service
uv run pytest tests/ -v -k "agent"

# 手动测试新提示词
uv run python scripts/test_prompt.py \
  --prompt v2 \
  --case "创建一个高优先级待办，标题是完成周报"
```

### 4.3 A/B 评估

对于提示词变更，建议做以下对比：

1. **准确率**：正确调用工具的比例
2. **参数准确率**：工具调用参数全部正确的比例
3. **回复友好度**：人工评分 1-5
4. **确认频率**：不必要的确认询问次数
